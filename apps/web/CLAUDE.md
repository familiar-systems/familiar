# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Scope

Covers `apps/web/` only: the Vite + React SPA (the authenticated app). Overrides the repo-root CLAUDE.md for anything under this directory. Rules it deliberately does **not** repeat live in the root CLAUDE.md: TypeScript strictness, the package dependency graph, the URL-contract apex table, and the cookie-scope rule.

## Today's surface

Built: the hub (campaign listing), the 4-step creation wizard (`features/onboarding/`), the campaign index (`/c/$campaignId`, which sends a wizard-incomplete campaign to the wizard and a complete one to its home page), and the home-page editor (`/c/$campaignId/t/$thingId`). Still ahead: the rest of the editor surface (this slice is paragraphs + headings only) and the AI system.

No global store and no React Query: routes fetch in `useEffect` and hold local `useState`. The only app-wide state is auth, carried in router context (below).

## Routing (TanStack Router, file-based)

Routes live in `src/routes/`; `routeTree.gen.ts` is generated by the `tanstackRouter` Vite plugin — never hand-edit it. `router.tsx` is the assembly point: it wires the tree into a typed singleton and registers it on the global `Register` interface so `<Link>` / `navigate` / `useRouteContext` are type-checked.

- **Pathless layouts.** A leading underscore means layout-only, no URL segment: `_authed.tsx` wraps every authed page, but `_authed/index.tsx` is still `/` and `_authed/settings.tsx` is `/settings`.
- **Branded ID params.** Brand IDs at the URL boundary in the route file's `parseParams`, with a co-located Zod schema — e.g. `parseParams: ({ thingId }) => ({ thingId: thingIdSchema.parse(thingId) })` (`_authed/c/$campaignId/t/$thingId.tsx`). Platform brands (`CampaignId`, `UserId`) come from `@familiar-systems/types-app`; campaign brands (`ThingId`, `BlockId`) from `@familiar-systems/types-campaign`. `parseParams` mints the brand; the typed API clients consume it.
- **basepath.** `createRouter({ basepath: import.meta.env.BASE_URL })`. Both `<Link>` and `useNavigate` honor it, so pass logical paths (`/settings`) and let the router prepend the preview prefix.

## Auth lives outside the router

`useAuth()` (`lib/auth.ts`) runs once in `App.tsx` and resolves `/me` **before** `<RouterProvider>` mounts. Mounting the router with a placeholder context would resolve `_authed`'s `beforeLoad` against `{kind:'unauthed'}` and bounce the user to `/login` before `/me` could refute it — so while the fetch is in flight, `App` renders a loading shell, not the router. This keeps the router synchronous and avoids a per-navigation `/me` refetch.

- `AuthState` is a sum type: `{ kind: "unauthed" } | { kind: "authed"; user: MeResponse }`. `useAuth` calls `hanko.validateSession()` (asks the Hanko backend, not just "is a token cached") then `client.GET("/me")`.
- The gate is `_authed.tsx`'s `beforeLoad`: it narrows `context.auth` to the `authed` variant, or throws `redirect({ to: "/login", search: { redirect: location.href } })` so deep links survive login. Child routes then read `user: MeResponse` with no nullability.
- Hanko SDK singleton is `lib/hanko.ts`. Bearer tokens are attached by openapi-fetch middleware on both typed clients (below). WebSocket auth can't use a header, so it rides the query string (see editor).

## API calls: use the generated typed clients, always

Every API call MUST go through the typed `openapi-fetch` clients in `src/lib/`:

- `src/lib/api.ts` exports `client` typed against `PlatformPaths` (platform server; baseUrl `apiBase`).
- `src/lib/campaigns-api.ts` exports `campaignClient` typed against `CampaignPaths` (campaign server; baseUrl is just `BASE_URL` — the service prefix `/campaign`, `/catalog` is baked into the OpenAPI path keys, not the base).

These types are generated from the Rust servers' utoipa OpenAPI specs via `mise run generate-types`. The TypeScript compiler checks every path, method, parameter, and body against the spec. **A route that doesn't exist or a wrong-shape body fails to compile.**

**Never write raw `fetch()` calls or string-literal URLs to the platform or campaign servers.** The typed clients exist precisely so that backend route changes are caught by `mise run typecheck` instead of surfacing as runtime 404s. If you need a new route, add it on the Rust side first, run `mise run generate-types`, then call it through the typed client.

The type pipeline: Rust struct (`#[derive(ToSchema)]`) -> utoipa OpenAPI JSON -> `tooling/openapi-codegen` -> `packages/types-campaign/src/openapi/...` (or `types-app` for platform). Component schemas resolve to ts-rs branded types so IDs stay distinct (`CampaignId` cannot be passed where `UserId` is expected).

## Preview-safe URLs

The app apex stacks a per-PR prefix in preview (`/pr-42/`); dev and prod are `/`. (Full apex table: root CLAUDE.md URL contract.) Vite's `base` comes from `VITE_BASE_PATH` (`vite.config.ts`) and is exposed at runtime as `import.meta.env.BASE_URL`.

**Build every internal URL through `lib/paths.ts`; never hardcode `/api/...` or a bare leading-`/` route** — a literal `/` bypasses the base and 404s in preview. Vite does not rewrite absolute paths in JSX or inline styles.

- `apiPath` / `campaignPath` / `catalogPath` — same-origin service calls (these back the typed clients' base URLs).
- `wsUrl` — absolute `ws://`/`wss://` for the campaign WebSocket, prefix preserved.
- `spaRoute` — browser navigation to an SPA route.
- `assetPath` — static asset served from `public/`.
- `siteLink` — cross-apex link to the marketing site (`VITE_SITE_URL`, which already includes the prefix in preview).

## Editor + Loro collaboration

The editor is split by concern: `@familiar-systems/editor` owns the shared TipTap schema and the Loro binding (`NODE_EXTENSIONS`, `BlockId`, `LoroExtension`, `readThingTitle`, `contentContainerId`); the React component, the WebSocket transport, and the doc/room lifecycle live here.

- `features/editor/loro-manager.ts` — `LoroClientManager`: **one campaign-scoped WebSocket multiplexing many CRDT rooms** (the "Repo" pattern from loro-extended). Every room is one generic ref-counted `RoomHandle<T>` (one doc, one join, a `select(doc) => T` view, subscribers); the ToC and per-Thing rooms are two call sites with different config, not two code paths. Room join/leave lives here, not in React effects, so the socket survives navigation between Things (no per-navigation handshake + full-snapshot re-pull). The underlying client is **constructed once and never swapped**; `connect()`/`close()` open it and schedule a **debounced teardown** (a remount's `connect()` cancels it), so StrictMode's connect→close→connect collapses to a no-op. The library's per-room `onStatusChange` is folded into the snapshot as a `reconnecting` state so the UI can show recovery rather than freeze.
- `features/editor/LoroManagerProvider.tsx` — constructs the manager **purely** (`useState`) and drives `connect()`/`close()` from a `useEffect`. The one-live-socket guarantee under StrictMode is the **manager's** (single never-swapped client + debounced teardown), not this effect's timing. Mounted once per campaign at the `/c/$campaignId` layout, **keyed by `campaignId`** so switching campaigns rebuilds the manager on the new socket URL. `useLoroManager()` reads it from context.
- `features/editor/HomeEditor.tsx` binds TipTap to a synced Loro doc. `BoundEditor` is split out so `useEditor` runs unconditionally (rules of hooks), only after sync. It passes **no `content` prop** — loro-prosemirror builds the document from the Loro container, and seeding TipTap content would race the Loro init.
- `features/editor/useThingDoc.ts` is a read-only subscription over the manager (`useSyncExternalStore` on a per-Thing snapshot, plus an effect that `acquireThing`/`releaseThing`). It maps the manager's generic `RoomSnapshot<LoroDoc>` into the `ThingDocState` sum type (`connecting | synced | reconnecting | error`). The ToC mirror is `features/toc/useToc.ts`: `useTocRoom()` (held once by the campaign layout) pins the room; `useToc()` maps the snapshot into `loading | ready | reconnecting | error`. Non-obvious invariants:
    - The manager creates a **fresh `LoroDoc` (random PeerID) per room** — never reuse a PeerID across concurrent writers.
    - A room exposes its view only after `waitForReachingServerVersion()`, so consumers mount against fully-synced content (no empty-doc flash).
    - StrictMode's double-mount is absorbed by the manager's single never-swapped client + debounced socket teardown + debounced room leave — not a per-mount `cancelled` guard.
    - Deep-linking straight to a Thing URL mounts the consumer's effect before the provider's `connect()` (React fires child effects first); `acquireThing` lazily opens the socket itself, so there is no gate to wait on.
    - The auth token rides the WS query string (`wsUrl(\`${campaignId}/ws?token=...\`)`) because the upgrade can't carry an `Authorization`header. **Known limitation:** the token is frozen at provider mount and the library reconnects to the same URL, so a token that expires before a socket drop+reconnect would fail the upgrade. The server gates auth at the upgrade and ignores per-room`JoinRequest.auth` (`apps/campaign/src/ws/{upgrade,connection}.rs`), so true reconnect-token-freshness is a cross-tier change (move WS auth off the upgrade URL), out of scope here.

## Testing

Three tiers, deliberately distinct (one term per concept — the old "e2e" specs were never end-to-end):

- **Unit (Vitest):** co-located `*.test.ts`; type-level tests are `*.test-d.ts`.
- **Integration (Playwright):** in `integration/` as `*.spec.ts`. The SPA runs in a real browser, but Hanko session validation, `/me`, and **every backend call** are mocked at the network layer (route interception). Asserts SPA-specific behavior — e.g. that navigation does no full-page reload and `Shell` stays mounted across it. `playwright.config.ts` spawns `pnpm dev` against `:5173`. Run via `mise run web:integration`; part of `mise run check`.
- **E2E (Playwright):** the genuine full stack — real platform + campaign + Loro WebSocket + SQLite, only the third-party Hanko edge stubbed. One spec, `e2e/smoke.spec.ts`, under `playwright.e2e.config.ts` (baseURL `http://app.localhost:18080`, **no** `webServer`). The `.mise/tasks/e2e` shell harness boots the whole stack on an isolated port block (13000/13001/15173/18080/19100, so it coexists with a running `mise run dev`), runs the spec, then SIGTERMs the campaign for a graceful flush and asserts the persisted SQLite via `apps/campaign/examples/smoke_assert_db`. The browser "login" is one seeded `hanko` cookie (the SDK's `getSessionToken()` reads it); the stub `/sessions/validate` is `e2e/support/mock-hanko.mjs`. Run via `mise run e2e` (needs `mise run setup:playwright` once); **not** in `check`.
- `vitest.config.ts` excludes both `e2e/` and `integration/` (those dirs are Playwright's, not vitest's).

## Build / serve

Multi-stage `Dockerfile` builds the static bundle and serves it from nginx. `nginx.conf`: SPA fallback (`try_files $uri $uri/ /index.html`), long-cache hashed `/assets/*`, no-cache `index.html`. The `VITE_*` vars (`VITE_BASE_PATH`, `VITE_HANKO_API_URL`, `VITE_SITE_URL`) are **build args baked into the bundle**, not runtime config.

## Dev notes

- No `server.proxy` in `vite.config.ts`: the dev reverse proxy is Caddy (`Caddyfile.dev`) on `:8080`, which exposes `app.localhost:8080/` as the SPA origin and forwards `/api` and `/campaign` to their backends. Vite itself serves on `:5173`.
- Plugin order matters: `tanstackRouter()` before `react()` (it transforms route files the React plugin then compiles); `wasm()` is required so Vite can load `loro-crdt`, which ships as WebAssembly.
