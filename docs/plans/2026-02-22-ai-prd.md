## 1. Purpose & Scope

familiar.systems's AI system is a **proposal engine** for the campaign graph. It processes session content, answers questions about the campaign world, and collaborates with the GM to expand and refine that world — but it never modifies the graph directly.

**The contract: AI proposes, the GM disposes.** Every AI output that would change the campaign graph materializes as a durable suggestion. The GM reviews and acts on each one. This is not a guideline — it is a hard architectural boundary. The AI has no write path to the graph that bypasses the suggestion layer.

**What the AI system is responsible for:**

- Processing session recordings and notes into structured journal entries
- Detecting entities (NPCs, locations, items, factions) mentioned in session content
- Proposing new entities, relationships, and content updates for GM review
- Detecting contradictions between new content and established canon
- Answering questions about the campaign for both GMs and players
- Collaborating interactively with the GM to plan, refine, and expand campaign content

**What the AI system is NOT:**

- **Not an author.** It drafts, proposes, and suggests. The GM's voice and judgment are final.
- **Not autonomous.** It cannot take action on the campaign graph without explicit GM approval.
- **Not a chatbot.** Every conversation has a focal point in the campaign and produces either answers or durable proposals — never idle conversation.

**Example — the boundary in practice:**

> After session 14, the AI processes the recording and identifies a new NPC: a street urchin named Pip. It does not create Pip in the campaign graph. It produces a `create_thing` suggestion with proposed blocks (name, description, first appearance). The suggestion sits in the review queue until the GM accepts it, edits it, or lets it auto-reject after ~14 days. Until the GM acts, Pip does not exist in the campaign world.

## 2. Campaign Context & Retrieval

The AI operates over the campaign graph — the interconnected structure of nodes (things), blocks (content), and edges (mentions and relationships) that represent the campaign world. Every AI interaction requires retrieving relevant context from this graph.

### 2.1 The Knowledge Base

The campaign graph serves as the AI's knowledge base. Every entity is a node, every connection is an edge, and content is composed of blocks that can be referenced and embedded across the graph. It includes:

- **Nodes** — things in the campaign world: NPCs, locations, items, factions, lore, player characters, arcs, sessions. A node can be a **prototype** (template) — a thing marked as a template that defines the default page layout and block structure for new things of that type. Creating an NPC "from the NPC prototype" clones that prototype's block structure as a starting point.
- **Blocks** — the atomic content unit. Everything inside a node is a block: text paragraphs, headings, stat blocks, images. Blocks are the grain at which content is referenced, embedded, and transcluded. A block from one node can be embedded live in another (transclusion), and blocks derived from audio carry references back to timestamps in the original recording.
- **Mentions** — block-to-node and block-to-block references: "this content points to that entity"
- **Relationships** — node-to-node semantic connections: "Kael frequents the Rusty Anchor"
- **Sessions** — the temporal spine: session metadata, journal entries, prep notes, raw sources

This is not a static document store. It's a live graph that grows with every session, and the AI must query it in real time during both interactive and batch workflows.

### 2.2 Status-Based Access Control

Every node, block, and relationship carries a **status**: `gm_only`, `known`, or `retconned`. Status filtering is applied **at the retrieval layer, before the AI sees any content.** This is an access control boundary, not a post-hoc filter.

| Role       | Sees                | Does NOT see                                                                |
| ---------- | ------------------- | --------------------------------------------------------------------------- |
| **GM**     | `gm_only` + `known` | `retconned` (excluded from active context, retrievable on explicit request) |
| **Player** | `known` only        | `gm_only` (structurally invisible — not redacted, absent)                   |

**Requirement:** When a player asks the AI a question, the AI must be unable to reference or reveal `gm_only` content — not because it chooses not to, but because the content was never in its context. The filter is applied before the AI sees anything, making information leakage structurally impossible.

**Example — player Q&A with status filtering:**

> A player asks: "What do we know about Clericman the Good?"
>
> Clericman's node is `known`, but it contains a `gm_only` block describing his secret allegiance to Murdergod, and a `gm_only` relationship connecting him to Murdergod. The AI retrieves only the `known` blocks — his public description, his role in the temple, his stat block. The secret allegiance does not exist in the AI's context. The player gets an answer that reflects exactly what their characters would know, with no indication that hidden content exists.

### 2.3 Focal Point Scoping

When the AI begins an interaction, its initial context is determined by where the user invoked it — the **focal point**.

| Focal point                      | Initial context                                                             |
| -------------------------------- | --------------------------------------------------------------------------- |
| Session page (after processing)  | Session transcript, extracted entities, journal draft, existing suggestions |
| Session page (before session)    | Recent session summaries, active plot threads, prep notes                   |
| Thing page (NPC, location, etc.) | The thing's blocks, relationships, all mentions across sessions             |
| Campaign overview                | Arcs, major entities, open contradictions, unresolved threads               |

The focal point determines what the AI loads automatically. It does not limit what the AI can access — the AI can still query the broader campaign graph via its tools.

### 2.4 Context Expansion via @-References

The GM can pull additional context into a conversation by `@`-referencing specific nodes or blocks.

**Example — @-reference expansion:**

> The GM opens the agent window from the Grimhollow page. The AI loads Grimhollow's blocks, relationships, and mentions. The GM types: "Flesh out Grimhollow's history, connecting it to @SilverCompact and @Kael."
>
> The AI now retrieves the Silver Compact and Kael nodes (blocks, relationships, mentions) and adds them to its working context. It can reason about all three entities and propose content that weaves them together.

### 2.5 Retrieval Capabilities

The AI must be able to retrieve campaign content in several ways:

- **Entity search** — find nodes by name, type, or description text (fuzzy matching)
- **Semantic search** — find blocks by meaning, not just keywords (vector similarity over content)
- **Graph traversal** — follow relationships from a node to discover connected entities
- **Session history** — retrieve summaries and journals from recent or specific sessions
- **Backlinks** — find all blocks that mention a given node (powered by the mention graph)

**Example — cross-session retrieval:**

> The GM asks: "What do we know about the Silver Compact across all sessions?"
>
> The AI searches for all mentions of the Silver Compact node across session journals, thing pages, and relationship edges. It synthesizes a summary spanning sessions 3, 7, and 12 — the Compact's first mention, their alliance with Ashenmoor, and the revelation of their true purpose. Each claim traces back to its source session.

**Example — relationship discovery:**

> The GM opens the agent window from Kael's page and asks: "Who is Kael connected to?"
>
> The AI traverses Kael's relationships: frequents the Rusty Anchor, employed by the Silver Compact, distrusts Tormund. It also surfaces mentions — Kael appears in the journal entries for sessions 4, 8, and 11. The GM gets a complete picture of Kael's connections, both explicit (relationships) and implicit (co-occurrences in journal content).

## 3. Tool System

The AI's behavior is determined by what tools it has access to. There are no mode toggles. When the AI has only read tools, it answers questions. When it has read and write tools, it can also produce suggestions. This distinction is structural — tool availability is set by the user's role, not by a UI toggle.

### 3.1 Read Tools

Available to all users (GM and players). Results are status-filtered by role before the AI sees them.

| Tool                      | Purpose                                      | Returns                                         |
| ------------------------- | -------------------------------------------- | ----------------------------------------------- |
| **Search entities**       | Find nodes by name, type, or description     | Matching nodes with summaries                   |
| **Get entity details**    | Retrieve a node's full content               | Blocks, status, metadata                        |
| **Get relationships**     | Retrieve a node's semantic connections       | Related nodes with labels and directions        |
| **Semantic search**       | Find content by meaning across all blocks    | Relevant blocks with source nodes and relevance |
| **Get session summaries** | Retrieve recent or specific session journals | Session narratives with key events              |

### 3.2 Write Tools

Available to GMs only. Every write tool produces a suggestion — never a direct graph mutation.

| Tool                      | Purpose                                                | Produces                                                              |
| ------------------------- | ------------------------------------------------------ | --------------------------------------------------------------------- |
| **Propose thing**         | Suggest creating a new entity                          | `create_thing` suggestion with prototype reference and initial blocks |
| **Propose blocks**        | Suggest new or updated content on an existing entity   | `update_blocks` suggestion                                            |
| **Propose relationship**  | Suggest a semantic connection between two entities     | `create_relationship` suggestion with label and optional inverse      |
| **Propose journal draft** | Suggest journal entry blocks for a session             | `journal_draft` suggestion                                            |
| **Flag contradiction**    | Flag conflicting content with references to both sides | `contradiction` suggestion                                            |

### 3.3 Behavioral Emergence

Tool availability determines behavior without explicit modes:

**Example — seamless Q&A to P&R transition:**

> A GM opens the agent window on Kael's page.
>
> **GM:** "What sessions mention Kael?"
> The AI uses read tools (search, session summaries) and answers with a list of sessions and relevant excerpts. This is Q&A behavior.
>
> **GM:** "Interesting. Flesh out his backstory — he should have a history with the Silver Compact before he arrived in town."
> The AI now uses write tools (propose blocks, propose relationship) alongside read tools. It proposes new blocks for Kael's backstory and a relationship "Kael → formerly affiliated with → Silver Compact." This is P&R behavior.
>
> No mode switch happened. The GM's intent shifted, and the AI's tool use shifted with it.

**Example — player-safe Q&A:**

> A player opens the agent window and asks: "Can you summarize what happened last session?"
>
> The AI has only read tools, and the retrieval layer filters to `known` content. It summarizes the published journal — the events the party experienced. Any `gm_only` blocks in the journal (e.g., "the party didn't notice the assassin in the rafters") are absent from the AI's context entirely. The player gets an accurate summary of what their characters know.

## 4. Suggestion System

Every AI output that would modify the campaign graph materializes as a **suggestion** — a durable, reviewable proposal. Suggestions are the universal output primitive for all AI write operations, whether from batch processing or interactive collaboration.

### 4.1 Suggestion Types

| Type                  | What it proposes                                                                    | Created when                                                                      |
| --------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `create_thing`        | A new node (NPC, location, item, etc.) with initial blocks, cloned from a prototype | AI detects a new entity in session content, or GM asks AI to create something     |
| `update_blocks`       | New or modified blocks on an existing node                                          | AI has additional information about an entity, or GM asks to flesh something out  |
| `create_relationship` | A semantic connection between two nodes, with label and optional inverse            | AI identifies a connection between entities in content, or GM asks to link things |
| `update_relationship` | A modification to an existing relationship (label change, inverse label, direction) | AI suggests normalizing a label, or GM asks to refine a connection                |
| `journal_draft`       | Proposed journal entry blocks for a session                                         | Session processing pipeline produces a draft from transcript and notes            |
| `contradiction`       | A flag that content conflicts with established canon, with references to both sides | AI detects inconsistency between new content and existing graph                   |

### 4.2 Lifecycle

```
AI generates suggestion
        ↓
Suggestion persisted immediately (pending)
        ↓
GM reviews:
  ├─ Accept  → content created in graph with gm_only status
  ├─ Edit    → GM modifies before accepting
  ├─ Reject  → suggestion marked rejected, no graph change
  ├─ Dismiss → suggestion marked dismissed, no graph change
  └─ Ignore  → auto-rejected after ~14 days
```

**Requirements:**

- **Immediate durability.** Suggestions are persisted the moment they are generated. If the GM closes the browser mid-conversation, every suggestion produced so far is waiting when they return.
- **Auto-rejection.** Suggestions not acted on within a configurable window (~14 days default) are automatically rejected. This prevents unbounded queue growth and embodies the "tolerant of neglect" principle. Auto-rejected suggestions remain visible in their conversation's history.
- **Default to gm_only.** When a suggestion is accepted, the resulting content (node, blocks, or relationship) is created with `gm_only` status. Promoting to `known` is always a separate, deliberate act.

### 4.3 Suggestion Batches

Related suggestions are grouped into a **SuggestionBatch** — the unit of review.

A batch represents a cohesive set of proposals from a single context. For example, a tavern scene in a session might produce a batch containing: one `create_thing` (the tavern), two `create_thing` (NPCs met there), three `create_relationship` (who frequents the tavern, who works there), and one `update_blocks` (a player character's notes updated with a new contact).

The GM can:

- Review and act on the entire batch at once (accept all, dismiss all)
- Expand the batch and act on individual suggestions
- Partially accept a batch (accept some, reject others)

**Example — SessionIngest batch:**

> After processing session 14's audio, the system produces two batches:
>
> **Batch 1 — Journal draft:** A `journal_draft` suggestion containing the proposed narrative for the session.
>
> **Batch 2 — Extracted entities and relationships:** Three `create_thing` suggestions (new NPCs: the dockmaster, a mysterious courier, a street urchin named Pip), two `create_relationship` suggestions (Pip → works for → the courier; the courier → visited → Northport docks), and one `update_blocks` suggestion (updating Northport's page with a new block about the docks district).
>
> The GM accepts the journal draft with minor edits, accepts Pip and the dockmaster, rejects the courier (it was actually a recurring NPC the AI didn't recognize), and edits the relationship to correct it.

### 4.4 Contradiction Detection

The `contradiction` suggestion type is unique — it proposes no graph mutation. Instead, it flags an inconsistency between new content and established canon, providing references to both sides.

**Example — contradiction detected during SessionIngest:**

> In session 14's transcript, the GM described the Silver Compact as "a merchant guild based in Ashenmoor." But in session 7's journal, they were described as "a covert military order." The AI flags this as a contradiction:
>
> _"The Silver Compact is described as a merchant guild in session 14, but was established as a covert military order in session 7 (see journal block). These descriptions may conflict."_
>
> The GM can then decide: retcon the earlier description, clarify that the "merchant guild" is a cover story (and add a `gm_only` block explaining this), or dismiss the contradiction as intentional ambiguity.

## 5. Agent Conversations

An **agent conversation** is a persisted record of an AI interaction — what was said, what was proposed, and what decisions were made. Conversations are the provenance for suggestions.

### 5.1 Conversations as Provenance

A suggestion without its originating conversation is context-free. "Create NPC: Mysterious Figure" means nothing on its own. The conversation explains: the GM was fleshing out Northport's underworld, asked for three shady contacts, and the AI proposed this NPC along with a rationale drawn from session 12's journal. **The conversation is the why.**

**Requirement:** Every suggestion must link back to the conversation that produced it. The GM must be able to trace from any suggestion to the full context of how and why it was proposed.

### 5.2 Conversation Initiators

| Initiator  | Trigger                             | Role     | Produces suggestions?          |
| ---------- | ----------------------------------- | -------- | ------------------------------ |
| **GM**     | GM opens agent window and types     | `gm`     | Yes (P&R) or No (Q&A)          |
| **Player** | Player opens agent window and types | `player` | No (Q&A only, read tools only) |
| **System** | SessionIngest pipeline completes    | `system` | Yes (batch proposals)          |

All three produce conversations with the same structure: messages, focal point, role, and suggestion references. The GM interacts with all three through the same agent window UI.

### 5.3 Conversation Lifecycle

- **Active** — has unresolved suggestions (at least one `pending`). Surfaces prominently — these represent outstanding decisions.
- **Resolved** — all suggestions acted on (accepted, rejected, dismissed, or auto-rejected). Fades from prominence but remains browsable in recent history.
- **Archived** — old resolved conversations. Still browsable but not surfaced in the UI unless explicitly searched for.

**Requirement:** Conversations with unresolved suggestions must be visually prominent. Once all suggestions are resolved, the conversation should remain accessible but not demand attention. Over time, resolved conversations transition to archived status.

### 5.4 "Continue From Here"

The GM can start a new conversation from any context where previous conversations exist. The new conversation inherits:

- The focal point's content
- Awareness of previous conversations on this focal point and their suggestion history
- Full access to the campaign graph via tools

**Example — continuing from a system conversation:**

> After SessionIngest produces a system conversation for session 14, the GM reviews the suggestions. Most are good, but the journal draft missed an important moment — the party's first encounter with Pip. The GM clicks "Continue from here" and types: "The journal is missing the scene where Pip pickpocketed Tormund. That's when we first met her. Work that into the journal and add a relationship between Pip and Tormund."
>
> The AI starts a new GM-initiated conversation with the session 14 context, the existing journal draft, and the system conversation's history. It proposes updated journal blocks and a new relationship: "Pip → pickpocketed → Tormund."

**Example — abandoning a dead-end conversation:**

> The GM starts a P&R conversation exploring Grimhollow's history. After a few exchanges, the direction isn't working. The GM can simply stop — the suggestions from this conversation are still durable and can be dismissed in bulk. Later, the GM can start a fresh conversation on the same focal point. The new conversation's AI can see that a previous conversation exists and what it proposed, but isn't constrained by it.

## 6. The Three Workflows

Three workflows, one interface. All three use the same agent window, the same suggestion model, and the same conversation persistence. They differ in trigger, duration, and tool availability.

### 6.1 SessionIngest (Batch Processing)

**Trigger:** The GM uploads session recordings and/or notes to a session page.

**What happens:**

1. **Transcription** — audio recordings are transcribed to text with timestamps
2. **Source merging** — transcript, GM notes, and player recollections are combined into a unified timeline with source markers
3. **Entity extraction** — the AI identifies entities mentioned in the content, resolves them against existing nodes in the campaign graph, and annotates references
4. **Journal drafting** — the AI produces a structured narrative draft from the annotated content, preserving entity references as clickable mentions
5. **Proposal generation** — the AI analyzes the journal to propose new entities, relationships, entity updates, and contradiction flags

**Output:** A system-initiated conversation on the session page with suggestion batches containing the journal draft and all proposals.

**Requirements:**

- Long-running (minutes to tens of minutes for a full session recording). Must not block the GM's use of the application.
- Must survive application restarts and deploys of other components.
- All suggestions persisted immediately as they are generated — no "all or nothing" completion requirement.
- Entity resolution must query the existing campaign graph to avoid duplicate proposals (e.g., recognizing "Kael" as an existing NPC rather than proposing a new one).

**Example — full SessionIngest flow:**

> The GM uploads a 3-hour recording and brief notes from session 14. The system begins processing:
>
> 1. Audio is transcribed, producing timestamped text
> 2. The transcript is merged with the GM's notes ("Pip should be a recurring NPC") and a player's submitted recollection ("We first went to the docks, then the tavern")
> 3. Entity extraction identifies 12 entity references: 8 match existing nodes (Northport, Kael, the party members), 3 are new (Pip, the dockmaster, a mysterious courier), and 1 is ambiguous
> 4. A journal draft is produced — a coherent narrative with embedded entity references
> 5. Proposals are generated: 3 new things, 5 new relationships, 2 updates to existing things, 1 contradiction flag
>
> The system conversation appears on session 14's page. The GM can review it immediately or come back days later — everything is durable. The whole review takes about 20 minutes.

### 6.2 Planning & Refinement (Interactive)

**Trigger:** The GM opens the agent window and starts a conversation.

**What happens:** The GM describes what they want. The AI retrieves relevant context from the campaign graph and responds with a mix of conversational text and structured suggestions. Responses stream in real time.

**Requirements:**

- Low latency — responses begin streaming within seconds
- Suggestions are persisted immediately as they are generated during the stream
- The AI has access to both read and write tools
- The GM can reference additional context with @-references mid-conversation

**Example — pre-session prep:**

> Before session 15, the GM opens the agent window on the upcoming session page. The AI loads context: where session 14 ended (the party at Northport docks), active plot threads (the Silver Compact investigation, Pip's introduction), and the GM's prep notes.
>
> **GM:** "The party is heading into the warehouse district next session. I need the warehouse to be a Silver Compact front. Give me the location and two NPCs who run it."
>
> The AI proposes:
>
> - `create_thing`: "The Iron Lantern Warehouse" — a location with blocks describing its public face (import/export business) and gm_only blocks about its true purpose
> - `create_thing`: "Harbormaster Venn" — an NPC who manages the warehouse publicly
> - `create_thing`: "Sable" — a Silver Compact operative who works in the shadows
> - `create_relationship`: Iron Lantern → "front for" → Silver Compact
> - `create_relationship`: Venn → "manages" → Iron Lantern
> - `create_relationship`: Sable → "operative of" → Silver Compact
>
> The GM accepts Venn and the warehouse, edits Sable's name to "Whisper," and adjusts the relationship labels.

**Example — world-building between sessions:**

> The GM opens the agent window from the Ashenmoor page. Ashenmoor has been mentioned in three sessions but has little detail beyond "a city to the east."
>
> **GM:** "Flesh out Ashenmoor. It should be the Silver Compact's base of operations — a trade city with a strong merchant council that the Compact secretly controls."
>
> The AI retrieves everything known about Ashenmoor and the Silver Compact, then proposes blocks describing the city's geography, government, and economy, plus relationships connecting the Compact to the merchant council. The GM reviews and accepts selectively.

### 6.3 Q&A (Interactive, Read-Only)

**Trigger:** A GM or player opens the agent window and asks a question.

**What happens:** The AI queries the campaign graph (filtered by the user's role) and streams a text answer with references to specific nodes, blocks, and sessions. No suggestions are produced.

**Requirements:**

- Low latency — responses begin streaming within seconds
- Content filtered by role: GMs see `gm_only` + `known`; players see `known` only
- Answers include references to source material (which session, which node, which block)
- No write tools available — structurally impossible to produce suggestions

**Example — player catching up:**

> A player missed session 13. They open the agent window and ask: "What happened in the last two sessions?"
>
> The AI retrieves the published journal entries for sessions 13 and 14 (filtered to `known` blocks only) and produces a summary. It mentions the key events, the NPCs encountered, and any significant changes to the party's situation. It does not mention the GM's notes about Pip being a Silver Compact informant (that block is `gm_only`).

**Example — GM investigating their own world:**

> The GM asks: "When did we establish that the Silver Compact operates in Northport? I can't remember if that's canon or just my notes."
>
> The AI searches across all sessions and the Silver Compact's node for mentions of Northport. It finds: a `known` journal block in session 7 where an NPC mentioned "Compact agents in the northern ports," a `gm_only` prep note for session 12 where the GM planned their presence, and the `gm_only` relationship "Silver Compact → operates in → Northport" added after session 14. The AI reports what's canon (the session 7 mention) versus what's GM-only (the prep note and relationship).

## 7. Mention & Relationship Extraction

The AI handles two kinds of connections between content, and they work fundamentally differently.

### 7.1 Mentions (Automatic, Exhaustive)

Mentions are block-to-node or block-to-block references. They represent "this content refers to that entity" — a factual, referential link.

**Requirements:**

- Mentions are **derived automatically** when the AI processes content — every entity reference in a journal block or thing page becomes a mention
- Mentions are **exhaustive** — if the content says "Jormag and Linnea went to the Rusty Anchor," that's three mentions
- Mentions carry **no label** — the connection is always "mentions"
- Mentions **inherit status** from the block they belong to — no independent visibility control
- Mentions power **backlinks**: "where is this entity mentioned across the campaign?"

### 7.2 Relationships (Proposed, Curated)

Relationships are node-to-node semantic connections. They represent "how these entities relate in the campaign world" — an interpretive, meaningful link.

**Requirements:**

- Relationships are **proposed by the AI, curated by the GM** — they always go through the suggestion layer
- Relationships carry a **freeform label** ("worships," "frequents," "allied with") and an optional **inverse label** ("is worshipped by," "is frequented by," "is allied with")
- Relationships have **independent status** — a relationship can be `gm_only` even when both connected nodes are `known`
- The relationship vocabulary is **emergent** — the GM does not predefine allowed labels. Labels are freeform text that accumulates over time.

### 7.3 The Distinction

**Mentions are the raw signal; relationships are the semantic interpretation.**

When the AI processes "Jormag and Linnea went to Northport":

1. **Mentions (automatic):** Three mentions are created — the journal block points to Jormag, Linnea, and Northport
2. **Relationships (proposed):** The AI proposes semantic connections — "Jormag → traveled to → Northport" and "Linnea → traveled to → Northport." These land in the suggestion queue for GM review.

Mentions are exhaustive and mechanical. Relationships are selective and require judgment — that's why they go through the suggestion layer.

### 7.4 Label Normalization

Over time, a campaign accumulates freeform relationship labels. The GM might use "works for," "employed by," and "serves" to mean the same thing.

**Requirement:** The AI should be able to identify clusters of semantically similar labels and suggest normalization — "These three labels appear to describe the same kind of relationship. Would you like to unify them?"

This is not a hard constraint at creation time. Labels remain freeform. Normalization is a periodic suggestion, not a gatekeeper.

**Example — label normalization:**

> After 20 sessions, the campaign has accumulated the labels "works for," "employed by," "serves," and "in service to" across 14 relationships. The AI identifies this cluster and surfaces a suggestion: "These four labels appear to describe the same kind of relationship (employment/service). Would you like to unify them under a single label?" The GM picks "serves" and the AI proposes `update_relationship` suggestions to normalize the others.

## 8. Constraints & Non-Requirements

### Hard Constraints

- **No direct graph mutation.** The AI produces suggestions. It never creates, updates, or deletes nodes, blocks, or relationships directly. The suggestion layer is the only write path.
- **Status filtering at retrieval.** Access control is enforced before the AI sees content. Player-facing AI contexts structurally cannot contain `gm_only` content.
- **Immediate durability.** Every suggestion is persisted the moment it is generated. No ephemeral proposals. No "save" action required from the GM.
- **Pluggable LLM provider.** The hosted instance uses managed API keys. Self-hosted instances bring their own provider and keys. The AI system must not depend on a specific LLM vendor.
- **Tolerant of neglect.** The system must remain useful when the GM doesn't review everything. Auto-rejection keeps the suggestion queue bounded. The system never piles up infinite unreviewed proposals.

### Non-Requirements (Explicitly Out of Scope)

- **Pre-defined ontology.** The system does not require or enforce a schema of allowed entity types or relationship labels. Structure emerges from play.
- **AI-driven automation.** The AI does not auto-accept its own suggestions, auto-publish content, or take any action without GM approval — even if its confidence is high.
- **Real-time collaboration with AI.** The AI does not co-edit documents in real time (that's the collaborative editing system's concern). The AI proposes blocks; the GM accepts them into the document.
- **Cross-campaign AI context.** The AI operates within a single campaign's graph. It does not query or learn from other campaigns, even on the same instance.

## 9. Open Questions

These are decisions that affect the AI system's requirements but have not yet been resolved.

### Retconned Content in AI Context

`Retconned` content is "no longer true, but visible." The current position: the AI excludes retconned content from active context retrieval but can reference it when explicitly asked ("what did we originally say about Kael's backstory?"). This needs validation — should the AI ever proactively surface retconned content (e.g., to warn about accidentally re-establishing something that was retconned)?

### Relationship Label Normalization

The AI should suggest normalizing semantically similar labels. Open: when does this happen? Options include periodic background analysis, during SessionIngest when new labels are created, or on-demand when the GM asks. The trigger and frequency need definition.

### Suggestion Dependencies

When the AI proposes a new entity and relationships for that entity in the same batch, what happens if the GM rejects the entity but doesn't review the relationships? The relationships reference a thing that was never created. Options: cascade rejection, warn on orphaned suggestions, or let auto-rejection handle it.

### Context Window Management

Long-running campaigns will accumulate more content than fits in any LLM's context window. How does the AI select and prioritize what context to include? The focal point and @-references provide explicit signals, but the AI must also make implicit decisions about what background context is most relevant. This strategy needs definition as campaigns grow beyond toy-example size.

### MCP Exposure

Should the campaign graph be exposed as an MCP server, allowing external AI tools to query campaign content? This could enable integration with AI coding assistants, custom scripts, or third-party tools — but it introduces a new access control surface that must respect status filtering.
